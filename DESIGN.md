# xcode-agent ΓÇö Design Spec (v0.1)

An agent-friendly CLI that gives AI agents first-class capabilities for iOS/macOS
development with Xcode ΓÇö the Apple-side counterpart to Google's Android agent dev
toolchain (see [`android-agent-toolchain.md`](android-agent-toolchain.md)).

## Goals

- **Primary differentiator: agent-friendly I/O.** Predictable, structured, low-token
  output an AI agent can drive without scraping logs.
- Native, dependency-free Swift binary ΓÇö nothing extra to install to run the CLI itself.
- Degrade gracefully when only Command Line Tools (not full Xcode) are present.

## Binary

- Name: `xcode` (no collision ΓÇö Apple ships `xcodebuild` / `xcode-select`, not `xcode`).
- Language: Swift (Swift Package Manager executable, no third-party dependencies).

## Project generator: Tuist

- `xcode create` writes a Tuist `Project.swift` manifest (+ sources) and runs
  `tuist generate` to produce a real `.xcodeproj`/`.xcworkspace`.
- `build` / `test` delegate to `tuist build` / `tuist test` when a Tuist project is detected.
- Rationale: real iOS app projects (not just Swift packages) from day one, with a
  declarative manifest. Chosen over XcodeGen for power/modularity.

## `create` command details

- **Output: a real `.xcodeproj`/`.xcworkspace`**, generated by Tuist from a scaffolded
  `Project.swift` manifest (the manifest is the source of truth; the project is a
  regenerable artifact, typically gitignored).
- **Templates** (`--template`):
  - `app` (default) ΓÇö a runnable SwiftUI application target.
  - `package` ΓÇö a Swift Package (no Tuist/Xcode project required).
- **Flags:** `--platform ios|macos` (default `ios`), `--bundle-id <id>`
  (default derived from name), `--no-generate` (scaffold the manifest + sources only,
  skip running `tuist generate`).
- **Default behavior: auto-generate.** `create` runs `tuist generate` after scaffolding
  so the result is immediately openable in Xcode. With `--no-generate`, or when `tuist`
  is absent, it writes the manifest + sources and exits with a `hint` to install/generate.

## Project detection order

Detection prefers the most specific project model first, because a Tuist project also
contains a `Package.swift` for dependencies:

1. Tuist manifest ΓÇö `Project.swift` / `Workspace.swift` / `Tuist/` directory
2. `*.xcworkspace`
3. `*.xcodeproj`
4. `Package.swift` (Swift Package)
5. none

## Agent-friendly I/O contract

- **`--json`** global flag ΓåÆ exactly one JSON object on **stdout**; all human logs and
  streamed build output go to **stderr**. Agents never parse log noise.
- **Response envelope:**
  ```json
  { "ok": true, "command": "simulator.list", "data": {}, "error": null, "hint": null }
  ```
- **Exit-code taxonomy** (stable, documented):
  | Code | Meaning |
  |---|---|
  | `0` | success |
  | `1` | runtime failure |
  | `2` | usage error |
  | `3` | environment not ready (e.g. needs full Xcode) |
  | `4` | required tool not found (e.g. tuist) |
- **Actionable errors:** every failure includes a `hint` with the literal command to fix it.
- **Introspection:** `xcode commands --json` self-describes every command + flags
  (MCP / AppFunctions-style capability discovery).

## Structured build/test output (v0.1: summary only)

- `build` / `test` stream raw tool logs to **stderr**.
- Final **stdout** JSON summary: `{ ok, exitCode, counts }`.
- `counts` derived via a cheap best-effort regex on the
  `Executed N tests, with M failures` line ΓÇö **no `.xcresult` parsing in v0.1**
  (deep parsing is a later milestone).

## Command set (v0.1)

| Command | Wraps | Notes |
|---|---|---|
| `doctor` | xcode-select, tuist, simctl | machine-readable readiness + remediation hints |
| `info` | toolchain versions | developer dir, Xcode, Swift |
| `commands` | ΓÇö | self-describing command/flag list (`--json`) |
| `create` | Tuist | writes `Project.swift` + runs `tuist generate` |
| `build` | tuist / xcodebuild / swift | dispatches by detected project kind |
| `run` | xcodebuild + simctl | build ΓåÆ install ΓåÆ launch on simulator |
| `test` | tuist test / xcodebuild test | summary counts |
| `simulator` | simctl | `list` / `boot` / `shutdown` |
| `skills` | `SKILL.md` discovery | `list` / `show` |
| `docs` | Apple Dev / HIG links | knowledge-base pointer |

## Graceful degradation

On a Command-Line-Tools-only machine (no full Xcode, no Tuist):

- **Fully functional:** `doctor`, `info`, `commands`, `skills`, `docs`, and SwiftPM
  `build` / `test`.
- **Clean exit 3/4 with install hints:** Tuist, simulator, and xcodebuild paths.

Fully functional on a real developer Mac with Xcode + Tuist installed.

## Later milestones (not v0.1)

- `.xcresult` deep parsing ΓåÆ per-test pass/fail, messages, durations.
- Knowledge-base index + search (beyond static links).
- `xcode update` via `xcodes` (manage Xcode/toolchain versions).
- MCP server fronting the same commands as tools for any agent.
- ADK-style in-app agent SDK (build agents *into* apps), ├á la ADK for Android.
